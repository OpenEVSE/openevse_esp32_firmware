## Background

After a WiFi dropout the ESP32 module frequently required a manual reboot to reconnect. The root cause was a cascade failure in `NetManagerTask::wifiOnStationModeDisconnected()`: the unconditional `WiFi.disconnect(true)` call aborted in-progress reconnection attempts, generating new disconnect events, which looped back and aborted each subsequent attempt indefinitely. Two additional gaps тАФ unhandled DHCP IP loss and a missed-disconnect-event blind spot тАФ could also strand the device in a connected-but-broken state.

All three fixes are confined to `src/net_manager.cpp`.

---

## Fix 1 тАФ `wifiStart()` reconnect storm (primary "requires reboot" bug)

**Function:** `NetManagerTask::wifiOnStationModeDisconnected()`

### Before
```cpp
  // Clear the WiFi state and try to connect again
  WiFi.disconnect(true);

  if(!isWiredConnected() && NetState::Connected == _state) {
    wifiStart();
  }
```

### After
```cpp
  if(!isWiredConnected() && NetState::Connected == _state) {
    // Update _state synchronously before posting the wifiStart message so that
    // any duplicate disconnect events queued behind this one (from the reconnection
    // process or from WiFi.disconnect()) don't each trigger their own wifiStart.
    _state = NetState::StationClientConnecting;
    // Full radio reset for a clean reconnection from a previously connected state.
    WiFi.disconnect(true);
    wifiStart();
  }
  // For StationClientConnecting / AccessPointConnecting: manageState() is already
  // driving the retry loop via _clientRetryTime тАФ don't call WiFi.disconnect(true)
  // here as it would abort a pending WiFi.begin() attempt.
```

### What changed
- `_state` is updated to `StationClientConnecting` **before** `wifiStart()` is called. Any duplicate disconnect events already queued (generated by the reconnection process or by `WiFi.disconnect()` itself) arrive and find the state is no longer `Connected`, so they skip both the `WiFi.disconnect(true)` call and `wifiStart()`.
- `WiFi.disconnect(true)` is now guarded inside the `Connected` branch тАФ it is only issued for a clean disconnect from a known-connected state. It is **not** called when already in `StationClientConnecting`, which would abort the pending `WiFi.begin()`.

## Fix 2 тАФ DHCP lease expiry / lost IP address

**Function:** `NetManagerTask::onNetEvent()` тАФ new `case ARDUINO_EVENT_WIFI_STA_LOST_IP`

### Added
```cpp
    case ARDUINO_EVENT_WIFI_STA_LOST_IP:
    {
      DBUGLN("Lost IP address");
      if(!isWiredConnected() && NetState::Connected == _state) {
        _state = NetState::StationClientConnecting;
        wifiStart();
      }
    } break;
```

### What changed
`ARDUINO_EVENT_WIFI_STA_LOST_IP` was previously unhandled. When a DHCP lease expired or the router withdrew the IP assignment the device stayed in `NetState::Connected` with no usable IP address indefinitely. The new handler transitions to `StationClientConnecting` and calls `wifiStart()` to obtain a fresh IP.

## Fix 3 тАФ Missed disconnect event watchdog

**Function:** `NetManagerTask::manageState()` тАФ `case NetState::Connected`

### Before
```cpp
    case NetState::Connected:
      if(millis() > _apAutoApStopTime)
      {
```

### After
```cpp
    case NetState::Connected:
      // Watchdog: if the disconnect event was missed, detect and recover here
      // rather than sitting in Connected state with no actual link.
      if(!isWifiClientConnected() && !isWiredConnected()) {
        DBUGLN("Watchdog: lost connection without disconnect event, reconnecting");
        _state = NetState::StationClientConnecting;
        wifiStart();
        break;
      }
      if(millis() > _apAutoApStopTime)
      {
```

### What changed
On every `manageState()` tick while in `Connected` state, `WiFi.isConnected()` is now checked (via `isWifiClientConnected()`). If the physical link is gone but no disconnect event was delivered (a known ESP-IDF race condition), reconnection is triggered immediately rather than waiting for an event that may never arrive.
